# pidory v0.7.2 — Permission Response Timeout Config

## English

### Highlights
- **Permission response timeout is now configurable**: Previously hardcoded at 5 minutes, the timeout for owner permission responses can now be adjusted via a new `[permission]` section in `config.toml`. Set `response_timeout_secs = 0` to disable the timeout entirely, letting owners respond after returning from meetings or stepping away.

### New Configuration

- **#372** — `[permission] response_timeout_secs` config option.

  Previously the permission response timeout was hardcoded at 300 seconds (5 minutes). If an owner was in a meeting, away from their desk, or asleep, the bot would auto-deny any permission prompts that arrived during that window, blocking long-running batch tasks.

  The timeout is now controlled via `config.toml`:

  ```toml
  [permission]
  # 0 = disabled (wait indefinitely). Default: 300 (5 minutes).
  response_timeout_secs = 0
  ```

  **When to use `response_timeout_secs = 0`**: Useful for batch tasks where the owner wants to review permission prompts upon returning rather than having them auto-denied. The bot will hold the permission future in memory until the owner responds or the session is terminated.

  **Backward compatibility**: If the `[permission]` section is omitted from `config.toml`, the behavior is unchanged from previous versions — 300-second timeout applies. No migration or config changes required.

  **⚠️ Operational note**: Setting `response_timeout_secs = 0` combined with unanswered permission prompts and a session that has not yet reached `idle_timeout_secs` will cause permission futures to accumulate in memory. For extended owner absence scenarios, review `[claude] idle_timeout_secs` (default: 7200 seconds) alongside this setting to ensure sessions eventually clean up.

### Migration from v0.7.1

No action required. The `[permission]` section is optional — existing `config.toml` files without it continue to use the 300-second default.

To opt in to indefinite waiting:

```toml
[permission]
response_timeout_secs = 0
```

## 한국어

### 주요 변경
- **권한 응답 timeout을 config로 조절 가능**: 기존에는 5분으로 하드코딩되어 있던 권한 응답 대기 시간을 이제 `config.toml`의 새 `[permission]` 섹션으로 자유롭게 설정할 수 있어요. `response_timeout_secs = 0`으로 설정하면 timeout을 완전히 비활성화해서, owner가 회의나 외출에서 돌아온 뒤 권한 요청을 처리할 수 있어요.

### 신규 설정

- **#372** — `[permission] response_timeout_secs` 설정 옵션.

  기존에는 권한 응답 대기 시간이 300초(5분)로 하드코딩되어 있었어요. 회의 중이거나 자리를 비웠거나 잠든 상태에서 권한 요청이 들어오면 봇이 자동으로 deny 처리해버려서, 긴 배치 작업이 차단되는 문제가 있었어요.

  이제 `config.toml`로 timeout을 조절할 수 있어요:

  ```toml
  [permission]
  # 0 = 비활성 (무한 대기). default 300 (5분)
  response_timeout_secs = 0
  ```

  **`response_timeout_secs = 0` 사용 시나리오**: owner가 돌아와서 권한 요청을 직접 처리하고 싶은 배치 작업에 유용해요. 권한 future는 owner가 응답하거나 세션이 종료될 때까지 메모리에 유지돼요.

  **Backward compat**: `config.toml`에 `[permission]` 섹션을 생략하면 이전 버전과 동일하게 300초 timeout이 적용돼요. 마이그레이션이나 config 변경이 필요 없어요.

  **⚠️ 운영 주의**: `response_timeout_secs = 0` + 권한 무응답 + `idle_timeout_secs` 미도달 상태가 누적되면 권한 future가 메모리에 잔존해요. 장시간 owner 부재 시나리오에서는 `[claude] idle_timeout_secs`(default: 7200초)도 함께 검토해서 세션이 결국 정리되도록 설정할 것을 권장해요.

### v0.7.1에서 마이그레이션

별도 조치 불필요. `[permission]` 섹션은 선택 사항이에요 — 기존 `config.toml`에 섹션이 없으면 기존과 동일하게 300초 default가 적용돼요.

무한 대기를 사용하려면 `config.toml`에 다음을 추가하면 돼요:

```toml
[permission]
response_timeout_secs = 0
```
