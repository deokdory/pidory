# pidory v0.7.0 — First Public Release

## English

### Highlights
- **First public release** of pidory under Apache-2.0 license
- **Database migration**: SQLite → PostgreSQL (breaking change)
- **Multi-user support** via Discord sender prefix
- **/update slash command** with pre-flight validation and pg_dump backup
- **i18n** with consistent friendly tone in Korean (해요체) and standard English locales

### Breaking Changes

> **P0-1 — Discord Developer Portal: enable `GUILD_MEMBERS` intent before upgrading.**
> pidory v0.7.0 requires the **Server Members Intent** for `@username` mention parsing (#346, #351).
> Without it the bot crashes on startup with `Disallowed intent(s). Library(4014)` in a restart loop.
>
> **How to enable:**
> 1. Open [Discord Developer Portal](https://discord.com/developers/applications) → your application → **Bot**
> 2. Under **Privileged Gateway Intents**, toggle **Server Members Intent** ON
> 3. Save changes, then proceed with the upgrade

> **P0-2 — `update.sh` does NOT update the systemd unit.**
> To upgrade from 0.6.x → 0.7.0, you **must** re-run `bash deploy/install.sh`.
> The 0.7.0 unit adds `Environment=HOME=...`, `EnvironmentFile=/etc/pidory/db.env`,
> `ExecStartPre=/usr/local/bin/pidory-migrate`, and extends `ReadWritePaths`.
> Running `update.sh` alone leaves the unit at the 0.6.x layout and causes
> `DATABASE_URL not set` failure on service start.

- **PostgreSQL required** (#304, #307): pidory now requires PostgreSQL 14+. SQLite is no longer supported.
  - Set `DATABASE_URL` environment variable
  - Run `sudo bash scripts/postgres-setup.sh` for automatic role/database/db.env setup
  - The `[database] path` field in `config.toml` is deprecated and ignored
- **License changed**: MIT → Apache-2.0. See [LICENSE](../../LICENSE) for the full text.

### New Features
- Discord sender prefix for multi-user threads (#316, #324) — distinguishes speakers in shared sessions
- /update command with pre-flight validation (#338) — blocks update if PostgreSQL setup is incomplete
- /update DB backup·restore via pg_dump/psql (#339) — automatic backup before migration
- claude_settings atomic editor (#295) — 7-layer concurrency protection for permission persistence
- 3-row permission button layout (#297, #298) — improved Discord UI for Allow/AlwaysAllow/Deny
- Tool kind cache (#306) — same-turn tool re-use auto-passes
- MCP tool permission exact-form rule (#310) — fixes infinite prompt loop
- Permission persistence with 5 protected prefixes (#318) — safer Always Allow scope
- /clear archived tombstone fix (#321)
- i18n cleanup with consistent 해요체 tone (#341)
- v0.7.0 misc bundle (#340): Fixes #288, #290, #293, #313

### Migration Guide

If you are upgrading from v0.6.x (SQLite):

1. **Backup existing SQLite database**:
   ```bash
   cp pidory.db pidory.db.backup
   ```

2. **Locate your legacy SQLite database**. The migrator looks for it in this order:
   - `PIDORY_LEGACY_DB` environment variable, or
   - `/var/lib/pidory/pidory.db` (default)

   If your SQLite DB is elsewhere, copy it to `/var/lib/pidory/pidory.db` or set
   `PIDORY_LEGACY_DB=/absolute/path/to/pidory.db` in the service environment.

3. **Re-run `install.sh` to update the systemd unit** (required — see P0-2 above):
   ```bash
   bash deploy/install.sh
   ```
   The following lines are new or changed in the 0.7.0 unit (applied automatically by `install.sh`):

```diff
 [Service]
+Environment=HOME=/home/<user>
 EnvironmentFile=-<project>/.env
+EnvironmentFile=/etc/pidory/db.env
+ExecStartPre=/usr/local/bin/pidory-migrate
-ReadWritePaths=<home>/claude <home>/.claude /tmp
+ReadWritePaths=<home>/claude <home>/.claude /tmp /var/lib/pidory
```

   If you skip this step, `DATABASE_URL` will not be injected and the service will fail to start.

4. **Install PostgreSQL** and run setup:
   ```bash
   sudo bash scripts/postgres-setup.sh
   ```
   This creates the `pidory` role/database and writes `/etc/pidory/db.env` (mode `0640`).

5. **Install the new `pidory-migrate` binary** (required before service restart):
   ```bash
   cargo build --release
   cargo build --bin pidory-migrate --features migrate --release
   sudo install -m 0755 target/release/pidory-migrate /usr/local/bin/pidory-migrate
   ```
   On service start, `ExecStartPre=/usr/local/bin/pidory-migrate` imports SQLite data
   into PostgreSQL (idempotent — re-runs are no-ops once PostgreSQL is populated).

6. **Restart the service** (or use `/update`):
   ```bash
   sudo systemctl restart pidory
   ```

7. **Verify**:
   ```bash
   systemctl status pidory
   journalctl -u pidory -f
   psql "$DATABASE_URL" -c 'SELECT count(*) FROM projects'
   ```

Migration is roll-forward only — there is no path back to SQLite.

### 마이그레이션 가이드

v0.6.x (SQLite)에서 업그레이드하는 경우:

1. **기존 SQLite DB 백업**:
   ```bash
   cp pidory.db pidory.db.backup
   ```

2. **레거시 SQLite DB 위치 확인**. migrator가 찾는 순서:
   - `PIDORY_LEGACY_DB` 환경변수, 또는
   - `/var/lib/pidory/pidory.db` (기본값)

   SQLite DB가 다른 경로에 있다면 `/var/lib/pidory/pidory.db`로 복사하거나,
   서비스 환경변수에 `PIDORY_LEGACY_DB=/absolute/path/to/pidory.db`를 설정해 주세요.

3. **`install.sh` 재실행으로 systemd unit 갱신** (필수 — P0-2 참고):
   ```bash
   bash deploy/install.sh
   ```
   `install.sh`가 자동으로 적용하는 0.6.x → 0.7.0 unit 변경 사항은 다음과 같아요:

```diff
 [Service]
+Environment=HOME=/home/<user>
 EnvironmentFile=-<project>/.env
+EnvironmentFile=/etc/pidory/db.env
+ExecStartPre=/usr/local/bin/pidory-migrate
-ReadWritePaths=<home>/claude <home>/.claude /tmp
+ReadWritePaths=<home>/claude <home>/.claude /tmp /var/lib/pidory
```

   이 단계를 건너뛰면 `DATABASE_URL`이 주입되지 않아 서비스 시작 실패해요.

4. **PostgreSQL 설치 + 셋업**:
   ```bash
   sudo bash scripts/postgres-setup.sh
   ```
   `pidory` role/database 생성 + `/etc/pidory/db.env` (권한 `0640`) 자동 작성.

5. **`pidory-migrate` 바이너리 설치** (서비스 재시작 전 필수):
   ```bash
   cargo build --release
   cargo build --bin pidory-migrate --features migrate --release
   sudo install -m 0755 target/release/pidory-migrate /usr/local/bin/pidory-migrate
   ```
   서비스 시작 시 `ExecStartPre=/usr/local/bin/pidory-migrate`가 SQLite 데이터를
   PostgreSQL로 임포트해요 (멱등 — PostgreSQL에 이미 데이터가 있으면 no-op).

6. **서비스 재시작** (또는 `/update` 사용):
   ```bash
   sudo systemctl restart pidory
   ```

7. **확인**:
   ```bash
   systemctl status pidory
   journalctl -u pidory -f
   psql "$DATABASE_URL" -c 'SELECT count(*) FROM projects'
   ```

마이그레이션은 단방향이에요 — SQLite로 롤백하는 경로는 없어요.
