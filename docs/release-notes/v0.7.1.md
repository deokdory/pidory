# pidory v0.7.1 — Migration Hotfix

## English

### Highlights
- **Critical hotfix for v0.7.0 migration data loss**: `pidory-migrate` previously defaulted to a fixed path (`/var/lib/pidory/pidory.db`) that did not match typical deployment layouts, causing the SQLite source to silently not be found and the migration to skip entirely. PostgreSQL started fresh, resulting in significant data loss for affected upgrades — the majority of sessions and projects could be missing depending on how the deployment laid out its SQLite database.
- **SQLite path resolution now uses a fallback chain** (env var → `config.toml` → default) with fail-loud behavior (exit code 2 with actionable guidance) if no source database is found.

### Fixes

- **#362** — `pidory-migrate` SQLite path resolution.

  Previously the migrator used only `/var/lib/pidory/pidory.db` as its source path. If the SQLite database lived elsewhere (e.g., `<repo>/pidory.db`, the common dev and self-hosted layout), the migrator would silently skip the entire migration and exit successfully. PostgreSQL was left empty, and the bot started creating new records from scratch.

  The resolution now follows a three-step fallback chain:
  1. `PIDORY_LEGACY_DB` environment variable — if set and non-empty, that path is used.
  2. `config.toml` `[database] path` field — if present and non-empty, that path is used (relative paths are resolved relative to the config file's directory, not the working directory).
  3. `/var/lib/pidory/pidory.db` — default.

  If the resolved path does not exist, `pidory-migrate` now exits with code 2 and prints an actionable error message listing all three sources and how to configure each. Silent skip has been removed.

  The idempotency behavior is unchanged: if PostgreSQL is already populated, the migrator exits cleanly without touching existing data.

- **#362 (continued)** — Wrapper script and deploy script verification steps.

  **`scripts/pidory-migrate.sh` (new)**

  A wrapper script that sources `/etc/pidory/db.env`, invokes `pidory-migrate`, classifies the exit code (0 = success, 1 = migration error, 2 = source DB not found, other = unexpected), and prints an actionable result message. In TTY environments the script pauses and waits for Enter before returning, so an operator running `install.sh` or `update.sh` interactively can read the output before the terminal scrolls past it. The script can also be re-executed manually at any time outside the systemd flow.

  **`deploy/install.sh` — step `[7/7] Pre-flight migration verification` (new)**

  After the binary is installed, `install.sh` calls the wrapper script. If `/etc/pidory/db.env` is missing the step warns and waits for the operator to press Enter to acknowledge before continuing. If the wrapper exits with a non-zero code the entire install aborts with exit 1. This replaces the previous behavior where a silent migration skip would go unnoticed until the bot came online with an empty database.

  **`deploy/update.sh` — step `[2/4] Checking systemd unit drift` (new)**

  Before replacing the binary, `update.sh` diffs the committed `deploy/pidory*.service` and `com.pidory.bot.plist` files against the previous commit using `git diff HEAD~..HEAD`. If any service file changed, the update aborts and instructs the operator to re-run `install.sh` instead. This prevents stale service units from running a new binary.

  **`deploy/update.sh` — step `[4/4] Pre-flight migration verification` (new)**

  Same wrapper call and pause pattern as `install.sh [7/7]`. Ensures that every `update.sh` run ends with a visible migration result rather than a silent skip.

### Migration from v0.7.0

If you upgraded from v0.6.x to v0.7.0 and your PostgreSQL appears empty or partially populated, the migration likely ran against the wrong path and silently skipped. Verify by comparing row counts against your SQLite backup:

```bash
# PostgreSQL current state
psql "$DATABASE_URL" -c 'SELECT count(*) FROM projects'
psql "$DATABASE_URL" -c 'SELECT count(*) FROM sessions'

# SQLite backup (adjust path as needed)
sqlite3 pidory.db.backup 'SELECT count(*) FROM projects'
sqlite3 pidory.db.backup 'SELECT count(*) FROM sessions'
```

If the counts differ and your SQLite backup is intact, you can recover:

1. **Stop pidory**:
   ```bash
   sudo systemctl stop pidory
   ```

2. **Restore your PostgreSQL state from backup** if you have a `pg_dump` snapshot from before the upgrade (or accept partial data loss if not).

3. **Point `pidory-migrate` to your SQLite database**. Use one of:
   - Set `PIDORY_LEGACY_DB=/absolute/path/to/pidory.db` in `/etc/pidory/db.env` or the service override.
   - Or ensure `config.toml [database] path` points to the SQLite file.

4. **Re-run `pidory-migrate`** — it is idempotent. It skips migration when PostgreSQL is already populated, so if you restored a clean PostgreSQL first, re-running is safe and will import the SQLite data.
   ```bash
   sudo /usr/local/bin/pidory-migrate
   ```

5. **Verify row counts** again, then restart:
   ```bash
   sudo systemctl start pidory
   ```

> **GUILD_MEMBERS intent**: If you did not enable the Server Members Intent in the Discord Developer Portal when upgrading to v0.7.0, do so now. See [v0.7.0 release notes](./v0.7.0.md) Breaking Changes (P0-1) for instructions.

### Migration from v0.6.x

Follow the [v0.7.0 Migration Guide](./v0.7.0.md). v0.7.1 fixes the path mismatch issue automatically — `pidory-migrate` now locates your SQLite database via `config.toml [database] path` even without the `PIDORY_LEGACY_DB` environment variable, as long as `config.toml` points to the correct file.

If `config.toml` does not have a `[database] path` entry and your SQLite database is not at `/var/lib/pidory/pidory.db`, set `PIDORY_LEGACY_DB` explicitly before starting the service.

## 한국어

### 주요 변경
- **v0.7.0 마이그레이션 데이터 손실 핫픽스**: `pidory-migrate`가 SQLite 소스 경로를 고정값(`/var/lib/pidory/pidory.db`)으로만 찾던 문제를 수정했어요. 이 경로가 실제 운영 배포 경로와 달라서 마이그레이션이 조용히 건너뛰어졌고, PostgreSQL이 빈 상태로 봇이 시작되면서 상당한 데이터 손실이 발생했어요 — SQLite DB 배포 위치에 따라 대부분의 세션과 프로젝트가 누락될 수 있어요.
- **SQLite 경로 resolution에 fallback chain 도입** (환경변수 → `config.toml` → 기본값). 소스 DB를 찾지 못하면 exit code 2와 함께 조치 안내를 출력해요 (silent skip 제거).

### 수정 사항

- **#362** — `pidory-migrate` SQLite 경로 resolution.

  기존에는 `/var/lib/pidory/pidory.db` 경로만 확인했어요. SQLite 데이터베이스가 다른 위치(예: `<repo>/pidory.db` — 개발 및 자체 호스팅 환경에서 흔한 구조)에 있으면 마이그레이션 전체를 조용히 건너뛰고 성공으로 종료됐어요. PostgreSQL은 빈 상태로 남고, 봇은 새 레코드를 처음부터 생성하기 시작했어요.

  이제 경로 resolution은 세 단계 fallback chain을 따라요:
  1. `PIDORY_LEGACY_DB` 환경변수 — 설정되어 있고 비어있지 않으면 그 경로를 사용해요.
  2. `config.toml`의 `[database] path` 필드 — 설정되어 있고 비어있지 않으면 그 경로를 사용해요 (상대 경로는 config 파일 위치 기준으로 resolve돼요).
  3. `/var/lib/pidory/pidory.db` — 기본값.

  resolve된 경로의 파일이 없으면 `pidory-migrate`는 exit code 2로 종료되고, 세 가지 설정 방법을 모두 안내하는 에러 메시지를 출력해요. Silent skip은 제거됐어요.

  멱등성 동작은 그대로예요: PostgreSQL에 이미 데이터가 있으면 마이그레이터는 기존 데이터를 건드리지 않고 정상 종료해요.

- **#362 (continued)** — 래퍼 스크립트 및 배포 스크립트 검증 단계.

  **`scripts/pidory-migrate.sh` (신규)**

  `/etc/pidory/db.env`를 source한 뒤 `pidory-migrate`를 실행하고, exit code를 분류(0 = 성공, 1 = 마이그레이션 오류, 2 = 소스 DB 없음, 기타 = 예상치 못한 오류)해서 조치 방법을 안내하는 결과 메시지를 출력하는 래퍼 스크립트예요. TTY 환경에서는 스크립트가 Enter를 누를 때까지 멈춰서, `install.sh` 또는 `update.sh`를 터미널에서 직접 실행하는 운영자가 화면이 스크롤되기 전에 결과를 읽을 수 있어요. systemd 흐름 바깥에서 언제든 수동으로 재실행하는 것도 가능해요.

  **`deploy/install.sh` — `[7/7] Pre-flight migration verification` 단계 (신규)**

  바이너리 설치 후 `install.sh`가 래퍼 스크립트를 호출해요. `/etc/pidory/db.env`가 없으면 경고를 출력하고 운영자가 Enter로 확인할 때까지 대기한 뒤 계속 진행해요. 래퍼가 0이 아닌 exit code로 종료되면 install 전체가 exit 1로 중단돼요. 기존에는 마이그레이션이 조용히 건너뛰어져도 봇이 빈 DB로 올라오기 전까지 알 수 없었는데, 이 단계가 그 silent skip을 차단해요.

  **`deploy/update.sh` — `[2/4] Checking systemd unit drift` 단계 (신규)**

  바이너리를 교체하기 전에 `update.sh`가 `git diff HEAD~..HEAD`로 커밋된 `deploy/pidory*.service` 및 `com.pidory.bot.plist` 파일의 변경 여부를 확인해요. 서비스 파일에 변경이 있으면 update를 중단하고 `install.sh`를 대신 실행하도록 안내해요. 오래된 서비스 유닛이 새 바이너리를 실행하는 상황을 방지해요.

  **`deploy/update.sh` — `[4/4] Pre-flight migration verification` 단계 (신규)**

  `install.sh [7/7]`과 동일한 래퍼 호출 + Enter 멈춤 패턴이에요. 모든 `update.sh` 실행이 silent skip 없이 마이그레이션 결과를 명시적으로 확인하며 끝나도록 해요.

### v0.7.0에서 마이그레이션

v0.6.x에서 v0.7.0으로 업그레이드했는데 PostgreSQL이 비어있거나 일부만 채워져 있다면, 마이그레이션이 잘못된 경로를 바라보고 조용히 건너뛴 가능성이 높아요. SQLite 백업과 row 수를 비교해서 확인해보세요:

```bash
# PostgreSQL 현재 상태
psql "$DATABASE_URL" -c 'SELECT count(*) FROM projects'
psql "$DATABASE_URL" -c 'SELECT count(*) FROM sessions'

# SQLite 백업 (경로는 맞게 조정)
sqlite3 pidory.db.backup 'SELECT count(*) FROM projects'
sqlite3 pidory.db.backup 'SELECT count(*) FROM sessions'
```

row 수가 다르고 SQLite 백업이 살아있다면 복구할 수 있어요:

1. **pidory 중지**:
   ```bash
   sudo systemctl stop pidory
   ```

2. **업그레이드 전 pg_dump 스냅샷이 있다면 PostgreSQL 상태를 복구**해요 (없으면 부분 손실 감수).

3. **`pidory-migrate`가 SQLite DB를 찾을 수 있도록 설정**해요. 방법 중 하나를 선택하세요:
   - `/etc/pidory/db.env` 또는 서비스 override에 `PIDORY_LEGACY_DB=/absolute/path/to/pidory.db` 설정.
   - 또는 `config.toml [database] path`가 SQLite 파일을 가리키도록 설정.

4. **`pidory-migrate` 재실행** — 멱등적이에요. PostgreSQL에 이미 데이터가 있으면 건너뛰므로, 빈 상태로 복구한 뒤 재실행하면 SQLite 데이터를 안전하게 임포트해요.
   ```bash
   sudo /usr/local/bin/pidory-migrate
   ```

5. **row 수 재확인** 후 재시작:
   ```bash
   sudo systemctl start pidory
   ```

> **GUILD_MEMBERS intent**: v0.7.0으로 업그레이드할 때 Discord Developer Portal에서 Server Members Intent를 활성화하지 않았다면 지금 해주세요. 절차는 [v0.7.0 릴리스 노트](./v0.7.0.md) Breaking Changes (P0-1)를 참고해요.

### v0.6.x에서 마이그레이션

[v0.7.0 마이그레이션 가이드](./v0.7.0.md)를 따라주세요. v0.7.1은 경로 mismatch 문제를 자동으로 해결해요 — `PIDORY_LEGACY_DB` 환경변수 없이도 `config.toml [database] path`를 통해 SQLite DB를 찾아요.

`config.toml`에 `[database] path` 항목이 없고 SQLite DB가 `/var/lib/pidory/pidory.db`에 없다면, 서비스 시작 전에 `PIDORY_LEGACY_DB`를 명시적으로 설정해주세요.
